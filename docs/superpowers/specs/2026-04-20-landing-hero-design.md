# Landing Hero Design

**Date:** 2026-04-20

**Goal:** показать неавторизованному пользователю отдельный лендинг-экран перед формой входа и сделать первую секцию максимально близкой к переданному референсу, но в брендинге проекта.

## Scope

В этой итерации реализуется только первый экран лендинга для гостя:

- отдельный hero вместо немедленного показа формы авторизации
- верхний header внутри hero
- левая текстовая колонка с брендингом и CTA
- правая крупная визуальная зона с серыми плейсхолдерами под будущие изображения
- переход на уже существующую форму логина по кнопке входа

Вне scope этой итерации:

- остальные секции лендинга
- новые роуты
- реальная навигация по пунктам header
- окончательные изображения и финальный маркетинговый текст

## User Flow

Если пользователь не авторизован, приложение сначала показывает landing state вместо текущей формы входа.

По клику на primary CTA в hero или на кнопку входа в header интерфейс переключается в login state на той же странице. При этом:

- URL не меняется
- текущая mock-auth логика не меняется
- после успешной авторизации пользователь попадает в существующий `HomePage`

## Visual Direction

Выбран вариант `A`: максимально близкая к референсу композиция.

Ключевые признаки:

- тёмный фон hero и тёмная шапка
- постерная композиция первого экрана
- две основные колонки
- слева крупный брендовый текст и кнопки
- справа высокий визуальный блок, который доминирует по массе
- серые заглушки вместо финальных картинок, чтобы позже подставить реальные ассеты

## Layout

### Header

Header располагается внутри landing hero и визуально повторяет пропорции референса:

- слева логотип проекта и название
- по центру или ближе к правой части три пункта навигации
- справа заметная кнопка входа

Header должен быть адаптивным:

- на desktop остаётся в одну строку
- на mobile допускается перенос или упрощение навигации, но без разрушения общего силуэта секции

### Hero Content

Основной экран делится на две зоны:

- левая зона: бренд, маленький логотип/бейдж, крупный заголовок, разделительная линия, две CTA-кнопки
- правая зона: один большой медиа-контейнер с серыми прямоугольными блоками-плейсхолдерами

Композиционная цель: первый экран должен читаться как плотный рекламный постер, а не как обычная форма или набор карточек.

## Content Rules

Контент должен быть в брендинге проекта, а не копировать текст с референса.

На первой версии допустимы временные project-branded тексты, но структура должна остаться такой:

- короткий брендовый kicker
- крупный headline в 2-3 строки
- primary CTA для перехода к логину
- secondary CTA как визуальная кнопка-заглушка без функциональной нагрузки или с безопасным действием

## Interaction

Landing screen должен поддерживать два перехода в форму входа:

- клик по кнопке входа в header
- клик по основной CTA в hero

Переход можно сделать простым переключением локального UI state без анимации на первой версии. Если анимация будет добавлена, она не должна мешать скорости и читаемости.

## Technical Approach

Текущий `AuthPage` остаётся входной точкой для неавторизованного состояния, но внутри него появляется локальный режим отображения:

- `landing`
- `login`

Рекомендуемая декомпозиция:

- вынести hero-лендинг в отдельный UI-компонент страницы auth
- оставить текущую форму логина отдельной частью
- хранить переключение режима рядом с `AuthPage`

Это позволит:

- не ломать существующую бизнес-логику авторизации
- не смешивать маркетинговый экран и форму входа в один длинный JSX-блок
- проще развивать лендинг дальше секциями

## Styling Constraints

- использовать существующий стек React + Tailwind
- сохранить визуальную точность референса по пропорциям и иерархии
- не строить секцию из карточной сетки
- не использовать фиолетовую тему по умолчанию вне тех мест, где она нужна для CTA
- серые плейсхолдеры должны быть явно читаемыми как временные зоны под изображения

## Responsiveness

Desktop — приоритетный режим для копирования референса.

На tablet/mobile:

- контент может перестроиться в одну колонку
- текстовый блок должен идти выше изображения
- CTA должны оставаться крупными и легко нажимаемыми
- header не должен ломать первый экран

## Testing

Нужно проверить:

- по умолчанию у гостя показывается landing, а не форма входа
- клик по CTA открывает текущую форму авторизации
- клик по header login тоже открывает форму авторизации
- логин после переключения работает как раньше
- hero корректно выглядит на desktop и mobile без переполнения

## Risks

- текущий `AuthPage` уже крупный, поэтому без декомпозиции он быстро станет слишком тяжёлым
- при слишком буквальном переносе desktop-композиции mobile может сломаться
- нельзя потерять существующую mock-auth функциональность ради лендинга

## Acceptance Criteria

- неавторизованный пользователь сначала видит landing hero
- секция визуально очень близка к референсу по композиции
- правая часть hero содержит серые плейсхолдеры под будущие изображения
- primary CTA и header login переводят на текущую форму входа
- существующая авторизация продолжает работать без регрессий
