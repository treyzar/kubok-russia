## 1. Суть задачи

Разработать **MVP веб-приложения** — платформу быстрых бонусных игровых комнат для VIP-аудитории Столото. Пользователь тратит бонусные баллы на участие в коротких real-time раундах и соревнуется за призовой фонд. Визуальная оболочка выбирается командой свободно (гонка, карточная игра, арена, колесо, сапёр и т.д.).

### Ключевые параметры

| Параметр | Значение |
|----------|----------|
| **Формат** | Веб-приложение с frontend, backend, хранением состояния и real-time обновлениями |
| **Сроки разработки** | 17–23 апреля (7 дней, стоп-кодинг 23.04 в 23:59) |
| **Команд участвует** | 33 команды → 20 финалистов |
| **Задача** | 1 задача от Столото |
| **Экспертных сессий** | 3 (18, 20, 22 апреля) |
| **Рекомендованный стек** | Java (backend) + PostgreSQL (СУБД) |
| **Авторизация** | Не обязательна |
| **Мобильная адаптация** | Не обязательна на данном этапе |
| **Денежные операции** | Вне рамок задачи |

## 2. Архитектура системы

Система состоит из 6 основных слоёв:

| Слой | Компонент | Ответственность |
|------|-----------|-----------------|
| **Frontend** | Любой стек | Список комнат, лобби, визуальный раунд, конфигуратор, журнал |
| **API Gateway** | REST API | Роутинг запросов, валидация входных данных |
| **Backend** | Java (рек.) | Бизнес-логика: комнаты, wallet, game engine, matchmaking |
| **Real-time** | WebSocket | Push-обновления всем участникам комнаты в реальном времени |
| **Хранилище** | PostgreSQL (рек.) | Пользователи, балансы, комнаты, история раундов |
| **Внешние сервисы** | RNG API | Генератор случайных чисел (опционально — внешний) |

### Модули Backend (обязательные)

#### Room service
CRUD комнат, управление состояниями (waiting / active / finished), изоляция раундов

#### Wallet / bonus service
Резервирование баллов при входе, списание после раунда, возврат при отмене, защита от фрода

#### Game engine
Запуск таймера, добавление ботов, применение буста, вызов RNG, определение победителя, начисление фонда

#### Matchmaking
Подбор комнаты по параметрам пользователя (цена, кол-во мест), фильтры, сортировка

#### WebSocket hub
Каналы на каждую комнату, рассылка изменений состава, таймера, фонда, статуса

#### Configurator API
CRUD параметров комнаты, валидация конфигурации, расчёт экономики

#### Round journal
Запись каждого раунда: участники, боты, бусты, фонд, победитель, дельты балансов

## 3. Алгоритм работы — 8 обязательных сценариев

Каждый сценарий оценивается дважды: в разделе «Готовность решения» (0–3 балла за реализацию) и в разделе «Удобство пользования» (0–3 балла за UX/UI). Если сценарий не реализован в коде — UX оценивается по дизайн-прототипу.

### С1. Список комнат и подбор игры
**Оценка: 0–3 балла × 2**

1. Пользователь открывает раздел быстрой бонусной игры.
2. Видит список доступных комнат с фильтрами и сортировкой.
3. Задаёт параметры подбора: цена входа, количество мест, дополнительные параметры.
4. Система предлагает подходящую комнату по выбранной конфигурации.

### С2. Вход в комнату, резервирование баллов и буст
**Оценка: 0–3 балла × 2**

1. Пользователь подтверждает участие → система резервирует бонусные баллы за вход.
2. До старта раунда пользователь может купить один буст за бонусные баллы.
3. Интерфейс показывает: занятые места, участников, ботов, фонд.
4. Отображается вероятность победы или понятное объяснение влияния буста.

### С3. Таймер, заполнение комнаты и real-time запуск
**Оценка: 0–3 балла × 2**

1. После входа первого пользователя запускается таймер ожидания (1 минута).
2. Реальные участники могут присоединяться к комнате в течение этого времени.
3. После завершения таймера свободные места заполняются ботами.
4. Все изменения отображаются в real-time: состав, занятые места, таймер, фонд, статус запуска.

### С4. Определение победителя и визуальный раунд
**Оценка: 0–3 балла × 2**

1. После старта backend получает: параметры комнаты, участников, места и бусты.
2. Backend определяет победителя (RNG / взвешенный алгоритм) и возвращает результат.
3. Пользователь видит визуальную оболочку раунда (гонка, арена, карты и т.д.).
4. Финал визуальной оболочки соответствует победителю из backend-логики.
5. После завершения — быстрый переход к следующей игре (повтор / похожая / более рискованная).

### С5. Конфигуратор и оценка параметров комнаты
**Оценка: 0–3 балла × 2**

1. Администратор открывает конфигуратор и меняет ключевые параметры.
2. Настраиваемые поля: количество мест, цена входа, параметры призового фонда.
3. Дополнительные настройки — если реализованы командой.
4. Система показывает, как конфигурация влияет на привлекательность для пользователя.
5. Система показывает выгодность для организатора или предупреждает о проблемных настройках.

### С6. Недостаточно бонусных баллов
**Оценка: 0–3 балла × 2**

1. Пользователь выбирает комнату или пытается купить буст.
2. Тестового баланса недостаточно для совершения действия.
3. Система корректно запрещает вход или покупку буста.
4. Показывает понятную причину отказа.
5. При необходимости предлагает выбрать другую комнату с более подходящими параметрами.

### С7. Невалидная конфигурация комнаты
**Оценка: 0–3 балла × 2**

1. Администратор задаёт неудачную или противоречивую конфигурацию.
2. Примеры: слишком высокий % фонда победителю, слишком дорогой буст, нереалистичная цена входа.
3. Конфигуратор обнаруживает такую ситуацию.
4. Показывает предупреждение, поясняет риск или не допускает сохранение.

### С8. Журнал раундов и прозрачность результатов
**Оценка: 0–3 балла × 2**

1. Эксперт или администратор открывает журнал раундов.
2. Видит историю проведённых игр: состав участников, боты, параметры комнаты.
3. Отображаются: наличие бустов, размер фонда, выбранный победитель, изменение балансов.
4. Система позволяет проверить, что результат соответствует backend-логике.
5. Ключевые действия корректно зафиксированы в истории.

## 5. Технические требования

### Обязательные требования

- Форма решения: веб-приложение.
- Frontend: технологии на усмотрение команды.
- Backend: технологии на усмотрение; рекомендуется Java.
- СУБД: на усмотрение команды; рекомендуется PostgreSQL.
- Полноценная регистрация и авторизация не обязательны.
- Real-time обновления состояния стола и системы обязательны.

### Дополнительные условия

- Не использовать платное и проприетарное ПО (риск нулевой оценки по критерию).
- Открытые библиотеки и open-source решения — приветствуются.
- Нейросетевые модели допускаются, если обоснованы и не создают излишних требований к ресурсам.
- Денежные платежи и пополнение баланса — вне рамок задачи.

### Требования к реализации

- Прототип развернуть локально или в облаке — с сетевым доступом для экспертов.
- Обеспечить доступ к решению в течение 10 рабочих дней после завершения соревнования.
- Разместить исходный код в репозитории GitHub.
- В README.md указать: инструкции по запуску, используемый стек, структуру проекта, демо-сценарии.
- Добавить документацию для организаторов: параметры игры и их влияние на систему.
- Эксперты должны иметь возможность пройти базовые сценарии без помощи команды.

## 7. Стратегические рекомендации

### Java + PostgreSQL = 20 бесплатных баллов
Два бинарных критерия в разделе «Архитектура» дают 20 баллов (или 0) без дополнительной работы. Выбирайте рекомендованный стек с самого начала.

### Конфигуратор и матчмейкинг — самый весомый блок (20 баллов)
Экономическая модель с реальными расчётами и матчмейкинг с видимыми ботами — именно это отличает сильные решения. Уделите этому приоритетное внимание.

### Дизайн-прототипы нереализованных сценариев
UX/UI оценивается даже для нереализованных сценариев — по прототипу. Набросайте Figma-прототипы для тех сценариев, которые не успеют войти в код.

### Реиграбельность — недооценённый критерий
10 баллов за петлю удержания (быстрый повтор, непрерывный игровой цикл). Экран результата должен вести прямо к следующей игре, а не тупиком.

### Журнал раундов — лёгкие баллы
Простая таблица с детальной историей каждого раунда даёт полные 3+3 балла. Реализуйте её в первую очередь после core-механики.
