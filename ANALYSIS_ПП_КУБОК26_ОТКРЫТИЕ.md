# Анализ документа «ПП Кубок 26_Открытие 17.04»

## 1. Что требуется по задаче

По материалам церемонии открытия нужно сделать **MVP веб-приложения** для быстрых игровых механик (VIP-аудитория), где пользователь:

1. Задает параметры подбора игровой комнаты.
2. Система подбирает комнату.
3. Резервируются бонусные баллы.
4. Запускается короткий real-time раунд.
5. Backend определяет победителя.
6. Пользователь видит понятную и прозрачную механику результата.

Ключевая суть: продукт должен давать быстрый эмоциональный опыт, быть прозрачным по правилам и экономически управляемым для организатора.

## 2. Обязательные сценарии, которые должны работать

Из документа явно следуют сценарии, которые эксперты должны пройти без помощи команды:

1. Список игровых комнат и подбор игры.
2. Вход в комнату, резервирование баллов и буст.
3. Таймер, заполнение комнаты и real-time запуск.
4. Определение победителя и визуальный раунд.
5. Конфигуратор и оценка параметров комнаты.
6. Обработка недостатка бонусных баллов.
7. Обработка невалидной конфигурации комнаты.
8. Журнал раундов и прозрачность результатов.

Это фактически ваш **минимальный acceptance set** для MVP.

## 3. Ограничения и формальные требования

1. Форма: только веб-приложение (mobile adaptation не обязательна).
2. Технологии frontend/backend/хранения можно выбирать самостоятельно.
3. Полноценная auth/registration не обязательна.
4. Желательно open-source стек, без обязательной зависимости от платного ПО.
5. Прототип должен быть доступен по сети для экспертов.
6. Доступ к решению должен сохраняться **10 рабочих дней после завершения** соревнования.
7. Исходный код должен быть в GitHub.
8. В `README.md` обязательно указать:
   - как запускать,
   - стек,
   - структуру проекта,
   - демонстрационные сценарии.
9. Отдельно нужна документация для организаторов по параметрам игры и их влиянию.
10. После стоп-кодинга правки в коде запрещены (риск дисквалификации).

## 6. Что нужно сделать в продукте (практический scope MVP)

### 6.1 Core функциональность (обязательно)

1. Каталог/листинг комнат с фильтрами.
2. Создание или подбор комнаты по параметрам.
3. Вход пользователя в комнату с проверкой баланса.
4. Резерв бонусов до старта раунда.
5. Таймер ожидания и auto-start раунда.
6. Расчет победителя на backend по прозрачным правилам.
7. Показ результата и изменения баланса.
8. История раундов с деталями расчета.

### 6.2 Обязательные edge-cases

1. Недостаточный баланс.
2. Невалидные параметры комнаты.
3. Выход игрока до старта/во время старта.
4. Таймауты при неполном наборе комнаты (отмена/перезапуск правила).
5. Дедупликация повторного join-запроса.

## 7. На что обратить особое внимание

### 7.1 Прозрачность определения победителя

Самый чувствительный блок для экспертов:

1. В правилах четко прописать формулу/алгоритм.
2. В раунде хранить входные параметры и seed.
3. В истории раунда показывать:
   - состав участников,
   - ставки/бусты,
   - как вычислен итог,
   - почему победил именно этот участник.

### 7.2 Согласованность баланса

1. Использовать транзакции для резервирования/списания/возврата.
2. Не допускать двойного списания при ретраях.
3. Иметь идемпотентные операции для join/start/settle.

### 7.3 Real-time стабильность

1. WebSocket/SSE для live-состояния комнаты.
2. Сервер — источник истины (не клиент).
3. Восстановление состояния при reconnect.
4. Сердцебиение/TTL комнаты, чтобы не копить «мертвые» раунды.

### 7.4 Демонстрационность и UX

1. Пользователь должен в 2-3 клика начать игру.
2. На экране до старта видны: цена входа, мест, фонд, эффект буста.
3. После раунда виден не только winner, но и «почему».

### 7.5 Процедурные риски

1. После 23:59 23.04.2026 нельзя менять код.
2. Нужен стабильно воспроизводимый запуск (1 команда/скрипт).
3. Доступ к демо не должен зависеть от локальной машины разработчика.

## Концепция механики для Кубка России

Предлагается реализовать систему игровых комнат с различными параметрами, в которых участники соревнуются между собой за внутренние бонусы.  
Каждая комната представляет собой отдельную игровую сессию с фиксированным числом участников (например: 1vs1, 10 человек, 20-30 человек). Создавать слишком большие комнаты (например, на 1000 человек) нецелесообразно, так как это ухудшает пользовательский опыт.

### Экономика и распределение выигрыша

Участники делают ставки (внутренние бонусы), формируя общий банк комнаты. Победитель получает не весь банк, а условно 80% от него. Остальная часть может использоваться для балансировки экономики или как комиссия системы.

### Механика вероятностей и бустеров

Изначально вероятность победы каждого участника равна и зависит от количества игроков (например, при 10 участниках - по 10% каждому).  
В игре предусмотрен бустер, который можно приобрести за внутренние бонусы. Он увеличивает вероятность выигрыша игрока. Например:

- было: 10 участников -> у каждого по 10%
- после покупки бустера одним игроком -> у него становится условно 20%, а вероятности остальных участников пропорционально уменьшаются

Ключевые ограничения:

- бустер можно купить только один раз
- использование бустера одним игроком влияет на вероятности всех остальных

### Формат участия

Игрок может занять сразу несколько мест в комнате (например, 2 или 3), тем самым увеличивая свою ставку и шансы на победу.

### Боты

Система должна предусматривать использование ботов для заполнения комнат и поддержания онлайна. Поведение ботов:

- боты не используют бустеры
- боты не находятся в комнате заранее - они подключаются динамически, примерно за минуту до старта
- пользователь не может заранее определить, есть ли в комнате боты
- модель поведения ботов и их процентное соотношение должны настраиваться (желательно через конфигурации)
- рекомендуется продумать стратегию, при которой боты могут иногда выигрывать (для поддержания баланса и реалистичности)

### Архитектура и нагрузка

Все расчеты (вероятности, выбор победителя и прочее) должны выполняться на стороне бэкенда.  
Система должна:

- работать в режиме реального времени
- выдерживать нагрузку порядка 100-200 RPS
- обслуживать до 100 000 пользователей в сутки

### Управление комнатами

Рекомендуется одновременно поддерживать порядка 30-40 активных комнат, в которые пользователи могут заходить.  
Важно:

- «хостинг комнаты» означает не физический сервер, а логическое создание комнаты в системе
- при необходимости количество комнат можно увеличивать (если появится больше типов)

Главная цель - обеспечить пользователю возможность в любой момент:

- зайти в подходящую комнату
- начать игру не более чем за 1 минуту

### Дополнительная сущность

В системе также вводится понятие «сущности» - это визуальный элемент (например, слово, значение или комбинация), который отображается пользователю на фронтенде. Он не влияет на механику игры и нужен исключительно для улучшения восприятия и интерфейса.
