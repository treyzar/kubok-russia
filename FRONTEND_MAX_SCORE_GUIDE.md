## 4. Что обязательно должно быть во фронте для максимума

## 4.1 Обязательные пользовательские сценарии (экспертный проход)

Нужно, чтобы эксперт мог пройти **8 сценариев** на UI (даже если данные моковые):

1. Список комнат и подбор игры.
2. Вход в комнату + резерв баллов + покупка буста.
3. Таймер, заполнение комнаты и запуск раунда в real-time отображении.
4. Определение победителя + визуальный раунд + быстрый переход к следующей игре.
5. Конфигуратор параметров комнаты + оценка последствий.
6. Недостаточно баллов (понятный отказ и подсказка).
7. Невалидная конфигурация комнаты (предупреждение и запрет сохранения).
8. Журнал раундов и прозрачность результата.

## 4.2 Что именно фронт обязан показывать прозрачно

На экранах должно быть явно видно:
- стоимость входа,
- количество мест,
- размер фонда,
- эффект буста,
- текущие вероятности участников,
- кто и почему победил (минимум: правила + входные данные + итог).

## 5. Модель «как брать баллы» по критериям (frontend-only)

## 5.1 Игровые механики и продуктовая модель (40)

Что делает фронт:
- понятный room-flow в 1-2 клика,
- таблица вероятностей в реальном времени интерфейса,
- визуально ясный раунд + финал + повтор,
- конструктор параметров комнаты с моментальной оценкой,
- понятные UX-реакции на ошибки (баланс, невалидные настройки).

Критично для баллов:
- у пользователя не должно оставаться вопроса «что происходит и почему».

## 5.2 Архитектурный вклад фронта (часть от 35)

Что показать в коде и презентации:
- `RoomShell` как переиспользуемый layout для любых мини-игр.
- slot-подход: игра подключается как `children`, а общие блоки едины.
- UI-контракты отделены от источника данных (`types.ts` + store/adapter).
- лёгкий переход mock -> backend за счёт стабильного интерфейса данных.

## 5.3 Качество кода и документация (24)

Нужно обеспечить:
- единый стиль компонентов и нейминг,
- малые, предсказуемые компоненты,
- README с запуском и демо-шагами,
- отдельный doc по параметрам игры и влиянию,
- doc по UI-сценариям для экспертов.

## 5.4 Удобство пользования (29)

Нужно закрыть:
- визуальная иерархия (что главное на каждом шаге),
- минимальная когнитивная нагрузка,
- читаемые состояния кнопок и ошибок,
- согласованный стиль всех экранов,
- базовая доступность: контраст, focus-visible, читаемые тексты.

## 5.5 Презентация MVP (10)

На стороне фронта нужно подготовить:
- скринкаст с непрерывным проходом сценариев,
- слайды с акцентом на UX-логику, прозрачность победителя и модульность.

## 8. Acceptance checklist для фронта (перед стоп-кодингом)

## 8.1 Функциональный чек

- [ ] Комнаты отображаются, фильтры/подбор работают (на моках).
- [ ] Вход в комнату меняет состояние и резервирует баланс.
- [ ] Буст покупается 1 раз, шанс обновляется визуально.
- [ ] При нехватке баллов действие блокируется и есть объяснение.
- [ ] Таймер/фазы раунда переходят автоматически.
- [ ] Победитель отображается корректно и прозрачно.
- [ ] Есть «Повторить» и цикл повторного участия.
- [ ] Есть журнал раундов.
- [ ] Есть конфигуратор и валидация невалидных настроек.

## 8.2 UX/визуальный чек

- [ ] Пользователь понимает текущий шаг без пояснений от команды.
- [ ] На каждом ключевом экране видны: цена, места, фонд, шансы, буст.
- [ ] Ошибки читаемы, короткие, предметные.
- [ ] Выбранный участник визуально выделен.
- [ ] Контраст и фокус-состояния не ломаются.

## 8.3 Технический чек

- [ ] `npm run build` проходит.
- [ ] Нет runtime-ошибок в консоли на базовом сценарии.
- [ ] Нет «мертвых» кнопок без поведения.

## 10. Скринкаст и защита: как показывать, чтобы брать максимум

Порядок скринкаста (без пауз и импровизации):
1. Список комнат + фильтр.
2. Вход в комнату и резерв баллов.
3. Покупка буста и обновление вероятностей.
4. Таймер и запуск раунда.
5. Финал, winner и объяснение.
6. Повторный вход за 1 клик.
7. Недостаток баллов.
8. Невалидная конфигурация и журнал раундов.

В каждом шаге проговаривать:
- «что пользователь делает»,
- «что система показывает»,
- «почему это прозрачно и проверяемо».

## 12. Главный принцип на финал

Один полностью прозрачный и безошибочный фронтовый сценарий приносит больше баллов, чем много частично сделанных экранов. 
Ставка на: **понятность механики + демонстрируемая полнота сценариев + безупречный экспертный проход**.
